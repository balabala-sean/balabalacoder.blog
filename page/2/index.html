synchronizer-内置锁

发表于 2016-09-05 | 分类于 java

java中的每个object可以作为一个lock, 而每个lock关联着一个条件队列（其实是2个，一个就绪队列，一个等待队列，只不过对于调用方而言，像是一个队列）。这个
条件队列中保存的元素是线程，java中关于内置锁和内置条件队列的3个方法：wait(),notify(),notifyAll()。这里的object lock和condition queue的“关联”关系，个人的理解是通过这个object lock，可以去释放条件队列中等待的线程，例如notify出队，notifyAll倒序所有出队。这和Condtion的await()、signal()、signalAll()用法基本一致(除了需要显式锁和对象锁的区别)。

volatile详解

发表于 2016-08-07 | 分类于 java

在java的线程中由于线程栈的存在，对于变量的每一次读写，并不是从主内存区(heap)中获取最新的值，这就可能造成线程A、线程B同时读取了某一个变量x的值，然后线程A对x做了修改，而线程B还在用线程栈中的x值，造成了最终的数据不一致问题。

volatile译为“不稳定的”，这就指示jvm，线程A、线程B每次使用x的时候，都去主内存中进行读取。这样在任何时候，多个线程之间看到的共享变量都是同一个值。内部的实现是禁用了高速缓存策略（cpu cache）

classLoader双亲委托

发表于 2015-12-03 | 分类于 jvm

双亲委托机制：
ClassLoader是通过双亲委托机制来搜索和加载类的，每个ClassLoader实例都有一个父加载器的引用（Parent ClassLoader）,他们是一个包含的关系，非继承关系。而且我们通过重载一些方法(findClass)，可以破坏原有的双亲委托机制。

有一个比较特殊的是BootClassLoader，它是内嵌在JVM中的，只有它是没有父加载器的，但是可以被用作其他ClassLoader的父加载器。

  • 当一个ClassLoader要加载某个*.class的时候，它会先把任务往上丢（丢给父类加载器），所以一层层的丢，最后丢到了BootClassLoader那里去了。
  • 那么BootClassLoader来试着加载，如果它没有加载成功，则返回给委托的发起者（就是顺序又反过来了，一层层的往下丢，丢给他儿子），也就是ExtensionClassLoade
  • 要是ExtensionClassLoader也没加载成，则继续往下丢，丢给AppClassLoader进行加载。
  • 如果AppClassLoader也没加载成，则返回给指定的文件系统或者网络等URL中加载此类。
  • 如果最后一层层往上丢，丢完之后又一层层往下丢，整个流程执行完了之后还没找着，就ClassNotFoundException了。
